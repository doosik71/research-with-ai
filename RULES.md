# 문서 생성 및 편집 규칙 (RULES)

## 1. 파일 및 폴더 명명 규칙

- **영문 소문자 사용**: 모든 폴더와 파일 이름은 영문 소문자를 기본으로 합니다.
- **공백은 '_' 로 대체**: 이름에 공백이 필요할 경우 반드시 언더바(`_`) 로 대체합니다.
  - 예시: `my folder` → `my_folder`
- **플랫폼 호환성**: 위 규칙은 Windows, Linux, macOS 모두에서 파일 시스템 호환성을 보장합니다.

## 2. 파일 형식 및 확장자

- **기본 형식**: 모든 문서는 Markdown(`.md`) 형식으로 저장합니다.
- **예외 형식**: 사용자가 명시적으로 요구하는 경우에만 아래와 같은 형식을 허용합니다.
  - 텍스트 파일: `.txt`
  - JSON 파일: `.json`
  - 기타 형식: 사전에 합의된 경우에 한해 허용

## 3. Lint 검사 및 오류/경고 처리

- **Lint 도구**: Markdown 파일은 `markdownlint`(또는 동등한 Lint 도구)를 사용해 문법 및 스타일을 검사합니다.
- **오류 처리**: Lint 결과에 오류가 발견되면 반드시 수정하고 커밋합니다.
- **경고 처리**: 경고는 가능한 한 최소화하고, 불가피한 경우 주석으로 이유를 명시합니다.

## 4. 수식 표기법

- **MathJax 사용**: 문서 내 수식은 MathJax 표기법을 따릅니다.
- **인라인 수식**: `$...$` 로 감싸서 인라인 수식을 작성합니다.
  - 예시: `Euler's formula is $e^{i\pi}+1=0$.`
- **블록 수식**: `$$...$$` 로 감싸서 블록 형태 수식을 작성합니다.
  - 예시:

    ```markdown
    $$
    \int_{a}^{b} f(x)\,dx = F(b) - F(a)
    $$
    ```

## 5. 기타 권장 규칙

- **헤더 구조**: Markdown 헤더는 `#` 레벨을 논리적으로 사용하고, 동일 레벨의 헤더는 연속해서 사용하지 않습니다.

## 6. 폴더 및 문서 관리 규칙

- **각 연구 주제별 폴더에는 반드시 `index.md` 파일이 존재해야 합니다.**
- `index.md` 파일은 해당 폴더 내의 문서 목록을 관리하며, 새로운 마크다운 문서를 추가할 때마다 목록을 업데이트합니다.
- 각 폴더 안에서는 연구 주제와 관련된 논문을 조사하고, 논문마다 별도의 마크다운 파일을 생성하여 내용을 요약·정리·설명합니다.
- `index.md` 의 기본 템플릿은 다음과 같습니다:

  ```markdown
  # Index
  
  - [문서1](doc1.md)
  - [문서2](doc2.md)
  ```

  위와 같이 각 문서에 대한 링크를 추가합니다.

---

- **지침 업데이트**: 별도로 언급하지 않는 한, 사용자가 기억하라고 요구하는 모든 지침은 이 `RULES.md` 파일에 추가·갱신한다.
  - 논문 제목과 저자명은 한국어로 번역하지 말고 원어를 그대로 사용한다.
- **연구 분야 작업**: 사용자가 특정 연구 분야에 대한 작업을 요청하면, 해당 분야 폴더(예: `continual_learning`)에서 작업을 수행한다.
- **논문 조사**: 연구 분야별 논문 조사는 기본적으로 **arXiv**에서 수행한다. 다른 출처가 필요하면 별도 언급한다.
- **논문 목록 파일**: 조사한 논문 목록은 해당 분야 폴더 안에 `paper_list.md` 파일로 저장한다. 각 항목은 다음 정보를 포함한다:
  - 저자명
  - 논문명
  - 발표연도
  - 논문 URL

  예시:

  ```markdown
  - **저자**: John Doe, Jane Smith
    **논문명**: Continual Learning with Elastic Weight Consolidation
    **연도**: 2020
    **URL**: https://arxiv.org/abs/2001.12345
  ```

위 규칙을 준수하여 연구 작업을 진행한다.

## 8. 논문 요약 규칙

- **요청 처리**: 사용자가 논문 요약을 요청하면 이는 일반적인 `summary` 요청이 아니라 **상세 분석 보고서** 작성 요청으로 간주한다.
- **검색**: 논문은 반드시 **arXiv**에서 검색한다. 검색 키워드와 논문 ID는 사용자가 제공하거나, 요청에 포함된 논문 제목을 기반으로 자동 검색한다.
- **PDF 획득**: arXiv 페이지에서 PDF 링크를 추출하고, 해당 PDF 파일을 다운로드한다.
- **텍스트 추출**: PDF에서 텍스트를 추출하여 본문, 초록, 서론, 방법, 실험, 결론 등 주요 섹션을 구분한다. 가능한 경우 표와 그림의 캡션도 포함한다.
- **상세 분석**: 추출된 텍스트를 바탕으로 다음 항목을 포함한 **포괄적인 요약 보고서**를 작성한다.
  1. **논문 메타데이터** – 저자, 제목, 발표 연도, arXiv ID, DOI(가능한 경우).
  2. **연구 배경 및 동기** – 문제 정의, 기존 연구와의 차별점.
  3. **핵심 기여** – 주요 아이디어, 이론적/실험적 기여.
  4. **방법론 요약** – 모델/알고리즘 구조, 핵심 수식(수식은 MathJax 사용).
  5. **실험 및 결과** – 데이터셋, 평가 지표, 주요 결과 표/그래프 설명.
  6. **한계 및 향후 연구** – 논문의 제한점 및 확장 가능성.
  7. **핵심 인사이트** – 실무 적용 시 고려사항 및 연구 커뮤니티에 미치는 영향.
- **읽지 않아도 이해 가능**: 보고서는 논문을 직접 읽지 않아도 전체 흐름과 핵심 내용을 파악할 수 있도록 충분히 상세히 기술한다.
- **파일 명명**: 요약 보고서 파일 이름은 논문의 **영어 제목**을 소문자, 언더스코어(`_`) 로 변환한 뒤 `.md` 확장자를 사용한다. 예시: `energy_based_models_for_continual_learning.md`.
- **저장 위치**: 해당 논문이 속한 연구 주제 폴더(예: `continual_learning`)에 저장한다.
- **index 업데이트**: 새로 만든 요약 파일은 해당 폴더의 `index.md` 에 링크를 추가한다.
- **버전 관리**: 요약 파일 추가·수정 시 커밋 메시지는 `docs: add summary <paper_title>` 형식으로 작성한다.

- **언어 정책**: 논문 요약 보고서는 **한국어**로 작성하되, 전문 용어(예: 모델명, 알고리즘명, 수식 등)는 **영어**를 사용할 수 있다. 이는 독자에게 친숙한 설명을 제공하면서도 핵심 기술 용어의 정확성을 유지한다.

## 9. 문법 체크 및 LaTeX 수식 변환 규칙

### 9.1 문법 체크

- 논문 요약 보고서를 생성한 뒤 반드시 `markdownlint`(또는 동등한 Lint 도구)로 문법 검사를 수행한다.
- Lint 결과에 오류가 있으면 즉시 수정하고, 수정 후 다시 Lint를 실행해 오류가 없음을 확인한다.

### 9.2 LaTeX 수식 변환

- 인라인 수식이 `\(` 와 `\)` 로 감싸져 있으면 이를 `$` 로 감싸는 형태로 변환한다.
  - 예시: `\(E = mc^2\)` → `$E = mc^2$`
- 블록 수식이 `\[` 와 `\]` 로 감싸져 있으면 이를 `$$` 로 감싸는 형태로 변환한다.
  - 예시: `\[\int_{a}^{b} f(x)dx\]` → `$$\int_{a}^{b} f(x)dx$$`
- 변환 작업은 자동 스크립트(예: 정규식 기반) 또는 수동 편집으로 수행하며, 변환 후에도 `markdownlint` 검사를 다시 실행한다.

### 9.3 적용 시점

- 위 규칙은 **논문 요약 보고서 생성이 완료된 직후**에 적용한다.
- 모든 새로운 요약 파일은 이 규칙을 준수해야 하며, 기존 파일도 필요 시 일괄 변환한다.
